## Current Proofs of Concept

The current user-facing pipeline prototype is
`heightmap_erosion_pipeline_poc.html`. It is a static, single-file WebGL2 applet:
no server, no Node, no packages, and no external scripts. The user chooses an
image file, the applet converts it to an 8-bit grayscale source heightmap, fits a
smooth `heightAndSlope` field, uploads that field as an `RGBA32F` texture, and
runs the ported ShaderToy erosion filter.

The pipeline currently displays:

- the source heightmap;
- the fitted B-spline height;
- an RGB visualization of `heightAndSlope`;
- the eroded height;
- the signed `ErosionFilter` delta encoded for RGBA display;
- the signed height delta channel as grayscale.

There is not yet a download/export button, persistent settings UI, or a polished
end-user layout. The current goal is still proof of correctness and pipeline
compatibility.

`phacelle_web_poc.html` is the earlier WebGL2 isolation test for Phacelle noise
and the erosion filter. It uses a synthetic height-and-slope texture rather than
a user-supplied heightmap, and it exposes erosion parameters as sliders.

`skeletonize_web_poc.html` is the earlier reconstruction experiment. It can
quantize an image, skeletonize terraces, and optionally fit the same spline model
to either skeleton pixels or dense source pixels. That experiment was useful for
comparing artifacts, but the current pipeline POC uses dense relaxed anchors at
every source pixel and does not run skeletonization.


## Original Shader Data Structures

These are manual notes about the data structures used inside the original
ShaderToy code, not necessarily the final browser applet data model.

`buffer_a` creates a `vec3` structure. It maps points \(p\) to `xyz` data, with
interpolation when the exact point is not present:

- `x` is height, normalized to `[0, 1]`;
- `y` and `z` are the directional derivatives of height;
- together, the three channels are `heightAndSlope`.

The code in `shader_a` applies a brush to edit this 3-vector in a way that keeps
height and slope consistent. The underlying position space \(p\) is a 2D space
spanning `[0, 1]` in each direction, so a slope of one would take us from the
bottom of the height range to the top across the width of the image.

`buffer_b` has a `heightmap` function and an erosion filter function.
`heightmap` takes a normalized position \(p\), grabs
`n = (height, x_gradient, y_gradient)` from `buffer_a`, passes \(p\) and `n`
into the erosion filter, uses the result to compute a simple tree simulation,
then packs the values using ShaderToy-specific data structure conventions.

`phacelle` is a `vec4` corresponding to the sine-wave cells generated by the
noise function:

- `x` is the magnitude of the phase-cell noise;
- `y` is the derivative of that value;
- `z` and `w` are a direction vector for the stripe;
- `yzw` combine into a gradient in the usual 2D space.


## HeightAndSlope Contract

The ShaderToy erosion filter expects a `vec3 heightAndSlope` at each normalized
position \(p = (p_x, p_y)\):

- `x` is height, normalized to `[0, 1]`;
- `y` is the derivative of height with respect to normalized `p.x`;
- `z` is the derivative of height with respect to normalized `p.y`.

The source image is read in canvas image coordinates, where `x` increases right
and `y` increases downward. The shader samples in texture coordinates where
`p.y` increases upward, so the upload step flips rows and negates the fitted
canvas-space `y` slope:

$$
\text{texture}(x, H - 1 - y)
= \left(h(x,y),\; s_x(x,y),\; -s_y(x,y),\; 1\right).
$$

This sign flip is a compatibility convention for the WebGL texture, not a
property of the spline fit itself.


## Dense Relaxed Multilevel B-Spline Fit

The current reconstruction method is a simplified dense version of the
multilevel B-spline approximation algorithm in Lee, Wolberg, and Shin,
["Scattered Data Interpolation with Multilevel B-Splines"](https://doi.org/10.1109/2945.620490)
([open PDF](https://cg.postech.ac.kr/papers/scatteredData.pdf)). For background
on why B-splines are a standard image-processing representation, see Unser,
["Splines: A Perfect Fit for Signal and Image Processing"](https://doi.org/10.1109/79.799930)
([author page and PDF](https://bigwww.epfl.ch/publications/unser9902.html)).

For each image pixel \(k\), the applet stores a dense anchor

$$
(x_k, y_k, z_k),
\qquad
z_k = \frac{0.2126 R_k + 0.7152 G_k + 0.0722 B_k}{255}.
$$

The fit uses five tensor-product cubic B-spline levels with grid spacings

$$
h_\ell = \max\left(4,\; \operatorname{round}\left(\frac{128}{2^\ell}\right)\right),
\qquad \ell = 0,\ldots,4,
$$

so the current spacings are \(128, 64, 32, 16,\) and \(8\) pixels.

For a point \((x,y)\) at a given level, define

$$
u = \frac{x}{h_\ell}, \quad v = \frac{y}{h_\ell}, \quad
i = \lfloor u \rfloor, \quad j = \lfloor v \rfloor, \quad
t_x = u - i, \quad t_y = v - j.
$$

The cubic basis functions used by the implementation are

$$
\begin{aligned}
B_0(t) &= \frac{(1-t)^3}{6},\\
B_1(t) &= \frac{4 - 6t^2 + 3t^3}{6},\\
B_2(t) &= \frac{1 + 3t + 3t^2 - 3t^3}{6},\\
B_3(t) &= \frac{t^3}{6}.
\end{aligned}
$$

A level's surface contribution is

$$
S_\ell(x,y)
= \sum_{a=0}^3 \sum_{b=0}^3
c^{(\ell)}_{i+a,j+b} B_a(t_x) B_b(t_y).
$$

The applet fits levels additively. Before fitting level \(\ell\), each anchor
has the current prediction

$$
\hat z^{(\ell-1)}_k = \sum_{m < \ell} S_m(x_k,y_k).
$$

The residual is relaxed by half an 8-bit height unit, which prevents the spline
from chasing quantization noise:

$$
r^{(\ell)}_k =
\operatorname{sign}(z_k - \hat z^{(\ell-1)}_k)
\max\left(\left|z_k - \hat z^{(\ell-1)}_k\right| - \epsilon,\; 0\right),
\qquad
\epsilon = \frac{0.5}{255}.
$$

For each anchor, the local tensor-product weights are

$$
w_{ab,k} = B_a(t_{x,k}) B_b(t_{y,k}),
\qquad
q_k = \sum_{a=0}^3 \sum_{b=0}^3 w_{ab,k}^2.
$$

Each nearby control point receives the Lee-Wolberg-Shin style weighted
contribution

$$
\phi_{ab,k} = \frac{w_{ab,k} r^{(\ell)}_k}{q_k}.
$$

When multiple anchors influence the same control point, the applet averages
those contributions with \(w^2\) weights:

$$
c^{(\ell)}_{i+a,j+b}
=
\frac{\sum_k w_{ab,k}^2 \phi_{ab,k}}
     {\sum_k w_{ab,k}^2}
=
\frac{\sum_k w_{ab,k}^3 r^{(\ell)}_k / q_k}
     {\sum_k w_{ab,k}^2}.
$$

After a level is fitted, its contribution is added to every anchor prediction
before the next, finer level is computed. Unlike the optimized version described
by Lee, Wolberg, and Shin, this POC does not refine the hierarchy into one
equivalent lattice. It keeps the five fitted levels and evaluates their sum
directly, which is simpler and adequate for a browser proof of concept.


## Slope Evaluation

The slope is computed analytically from the fitted spline, not by finite
differences on the rendered height image. The basis derivatives are

$$
\begin{aligned}
B'_0(t) &= \frac{-3 + 6t - 3t^2}{6},\\
B'_1(t) &= \frac{-12t + 9t^2}{6},\\
B'_2(t) &= \frac{3 + 6t - 9t^2}{6},\\
B'_3(t) &= \frac{3t^2}{6}.
\end{aligned}
$$

At one level,

$$
\frac{\partial S_\ell}{\partial x}
=
\sum_{a=0}^3 \sum_{b=0}^3
c^{(\ell)}_{i+a,j+b}
\frac{B'_a(t_x)}{h_\ell} B_b(t_y),
$$

and

$$
\frac{\partial S_\ell}{\partial y}
=
\sum_{a=0}^3 \sum_{b=0}^3
c^{(\ell)}_{i+a,j+b}
B_a(t_x) \frac{B'_b(t_y)}{h_\ell}.
$$

Those derivatives are initially in height units per pixel. The shader's
coordinate system spans `[0, 1]` across the image, so the POC scales them into
height units per normalized image unit:

$$
s_x = (W - 1) \sum_\ell \frac{\partial S_\ell}{\partial x},
\qquad
s_y = (H - 1) \sum_\ell \frac{\partial S_\ell}{\partial y}.
$$

This scaling matches the original shader convention: a slope of one means the
height would change by the full normalized height range over the full normalized
width or height of the image.


## WebGL Erosion Pass

The applet uploads the fitted field as an `RGBA32F` texture. It uses nearest
sampling because each texel already contains the evaluated height and derivative
for one source pixel. Rendering into the debug canvases still uses display
encoding, so the visualized RGBA channels should not be confused with the
floating-point `heightAndSlope` input texture.

The current erosion parameters are hard-coded in `renderErosionPass`:

- `erosionStrength = 0.22`;
- `erosionScale = 0.15`;
- `gullyWeight = 0.5`;
- `detail = 1.5`;
- `octaves = 5`;
- `cellScale = 0.7`;
- `normalization = 0.5`.

The shader's `fadeTarget` is currently derived from altitude:

$$
\text{fadeTarget} =
\operatorname{clamp}\left(\frac{h - 0.45}{0.15}, -1, 1\right).
$$

This is a pragmatic standalone choice for PNG-in/PNG-out testing. It is not
known to match Leveller's application-side conventions exactly.


## Implementation References and Open Questions

Leveller is worth keeping in mind as a reference implementation. Its resource
page links C++ source for an Erode/Phacelle plugin based on Rune's shader.
Checked 2026-05-25: the public source zip contains only the Phacelle/Erosion
kernel plus a small GLSL type shim. The exported function has the signature
`ErosionFilter(settings, p, heightAndSlope, fadeTarget)`, so the plugin source
expects `heightAndSlope.yz` to already contain slope; it does not show how
Leveller computes slope from the underlying heightfield.

What remains unknown is Leveller's application-side slope convention. Public
Leveller pages mention slope maps, curvature maps, normal maps, and a slope
evaluation kernel, which suggests this lives in broader app/render/cache code,
not in the published Phacelle kernel. For this standalone PNG-in/PNG-out port,
the B-spline derivative convention above is our own compatibility choice rather
than something copied directly from Leveller.

Future performance reference: revisit MangoButtermilch's browser widget/source
for `3d-volume-generator-web` / Texturinator
(https://github.com/MangoButtermilch/3d-volume-generator-web). Its terrain
height-map tool is not trying to solve exactly the same input-output widget
problem, but its rendering of Rune's erosion filter to an image appears much
more performant than the current local widget. Check how its WebGL/render target
path is structured before doing any performance cleanup here.
